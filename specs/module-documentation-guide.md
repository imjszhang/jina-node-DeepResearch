# 模块文档规范指南

> 创建时间: 2025-12-20  
> 最后更新: 2025-12-20

## 概述

本指南定义了项目中模块级文档的标准格式和写作规范。所有模块应遵循此规范编写 `README.md` 和 `CHANGELOG.md` 文档，以确保文档的一致性、可读性和可维护性。

## 文档结构要求

每个模块目录下应包含以下文档：

```
module-name/
├── README.md           # 模块说明文档（必需）
├── CHANGELOG.md        # 更新日志（必需）
├── index.js            # 模块入口
└── ...                 # 其他源文件
```

---

## README.md 规范

### 基本结构

README.md 应按以下顺序包含各个章节：

| 章节 | 必需 | 说明 |
|------|------|------|
| 标题和元信息 | ✅ | 模块名称、创建/更新时间 |
| 概述 | ✅ | 简要描述模块用途 |
| 核心功能 | ✅ | 功能特性列表 |
| 架构设计 | ✅ | 目录结构和组件关系 |
| 组件说明 | ✅ | 各组件的详细说明 |
| 配置说明 | ⚪ | 配置文件格式和字段说明 |
| API 接口 | ⚪ | HTTP/WebSocket 接口文档 |
| 使用示例 | ✅ | 代码和命令示例 |
| 工作流程 | ⚪ | 流程图或时序图 |
| 注意事项 | ⚪ | 使用限制和注意点 |
| 依赖模块 | ⚪ | 外部依赖说明 |

### 详细规范

#### 1. 标题和元信息

```markdown
# 模块名称 模块中文描述

> 创建时间: YYYY-MM-DD  
> 最后更新: YYYY-MM-DD
```

**要求**:
- 标题使用英文模块名 + 中文描述
- 创建时间和更新时间使用 ISO 日期格式 (YYYY-MM-DD)
- 每次重大更新需同步修改"最后更新"时间

#### 2. 概述

```markdown
## 概述

[模块名称] 是一个 [功能描述]，用于 [使用场景和目的]。
```

**要求**:
- 使用一到两句话简洁描述
- 说明模块的主要用途和适用场景
- 避免过于技术化的术语

#### 3. 核心功能

```markdown
## 核心功能

- 📊 **功能名称** - 功能简述
- 🔍 **功能名称** - 功能简述
- 📈 **功能名称** - 功能简述
```

**要求**:
- 使用 emoji + 粗体功能名 + 简述 的格式
- 每个功能一行，保持简洁
- emoji 应与功能语义相关

**推荐 emoji 对照表**:

| 功能类型 | 推荐 Emoji |
|---------|-----------|
| 数据/统计 | 📊 📈 📉 |
| 搜索/查询 | 🔍 🔎 |
| 配置/设置 | ⚙️ 🔧 |
| 日程/时间 | 📅 ⏰ 🕐 |
| 报告/文档 | 📝 📄 |
| 网络/通信 | 🌐 📡 |
| 安全/验证 | 🔐 🔒 ✅ |
| 监控/追踪 | 👁️ 📍 |
| 存储/数据库 | 💾 🗄️ |
| 通知/消息 | 📢 💬 🔔 |

#### 4. 架构设计

```markdown
## 架构设计

\`\`\`
moduleName/
├── index.js                 # 模块入口，说明
├── ComponentA.js            # 组件A说明
├── ComponentB.js            # 组件B说明
└── html/                    # 子目录说明
    ├── index.html
    └── index.js
\`\`\`
```

**要求**:
- 使用树形结构展示目录
- 每个文件/目录后添加简要注释
- 注释使用 `# ` 开头，保持对齐

#### 5. 组件说明

```markdown
## 组件说明

### 组件名称

组件的职责描述：

- 功能点1
- 功能点2
- 功能点3

**状态枚举**（如适用）:
- \`state1\` - 状态说明
- \`state2\` - 状态说明
```

**要求**:
- 每个主要组件使用三级标题
- 先描述职责，再列举功能点
- 如有状态机或枚举，需明确列出

#### 6. 配置说明

```markdown
## 配置说明

### 配置文件位置

\`\`\`
{base_dir}/config/config-file.json
\`\`\`

### 配置格式

\`\`\`json
{
  "field1": "value1",
  "field2": {
    "nestedField": "value"
  }
}
\`\`\`

### 配置字段

| 字段 | 类型 | 必需 | 默认值 | 说明 |
|------|------|------|--------|------|
| \`field1\` | string | ✅ | - | 字段说明 |
| \`field2\` | object | ❌ | {} | 字段说明 |
```

**要求**:
- 配置路径使用变量占位符（如 `{base_dir}`）
- 提供完整的 JSON 示例
- 使用表格列出所有字段
- 必需字段使用 ✅，可选使用 ❌
- 类型包括：string, number, boolean, array, object

#### 7. API 接口

```markdown
## API 接口

### HTTP API

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | \`/api/module/resource\` | 获取资源 |
| POST | \`/api/module/resource\` | 创建资源 |
| PUT | \`/api/module/resource/:id\` | 更新资源 |
| DELETE | \`/api/module/resource/:id\` | 删除资源 |

### WebSocket 事件

**命名空间**: \`/module-name\`

**服务端事件**:
- \`eventName\` - 事件说明

**客户端事件**:
- \`eventName\` - 事件说明
```

**要求**:
- HTTP API 使用表格格式
- 路径参数使用 `:param` 格式
- WebSocket 需说明命名空间
- 区分服务端和客户端事件

#### 8. 使用示例

```markdown
## 使用示例

### 代码集成

\`\`\`javascript
const { setup, getService } = require('./moduleName');

// 初始化
const service = setup(options);

// 启动服务
await service.start();

// 使用服务
const result = await service.doSomething();

// 停止服务
await service.stop();
\`\`\`

### API 调用示例

\`\`\`bash
# 获取资源
curl http://localhost:3000/api/module/resource

# 创建资源
curl -X POST http://localhost:3000/api/module/resource \
  -H "Content-Type: application/json" \
  -d '{"field": "value"}'
\`\`\`
```

**要求**:
- 提供代码和命令行两种示例
- 代码示例应展示完整的生命周期
- 命令行示例添加注释说明

#### 9. 工作流程

```markdown
## 工作流程

\`\`\`
┌─────────────────┐
│  步骤1          │
└────────┬────────┘
         ▼
┌─────────────────┐
│  步骤2          │◀──── 说明
└────────┬────────┘
         ▼
    ┌────┴────┐
    │ 条件判断 │
    └────┬────┘
     是 │    │ 否
        ▼    ▼
   ┌────┐  ┌────┐
   │结果A│  │结果B│
   └────┘  └────┘
\`\`\`
```

**要求**:
- 使用 ASCII 图形绘制流程图
- 标注条件分支和说明
- 保持对齐和美观

**常用符号**:
- 方框：`┌──┐ └──┘`
- 箭头：`▼ ▲ ◀ ▶`
- 连接：`│ ─ ┬ ┴ ├ ┤`
- 角：`┌ ┐ └ ┘`

#### 10. 注意事项

```markdown
## 注意事项

1. **要点标题**: 详细说明
2. **要点标题**: 详细说明
3. **要点标题**: 详细说明
```

**要求**:
- 使用有序列表
- 每条以粗体要点开头
- 简明扼要说明限制或注意点

#### 11. 依赖模块

```markdown
## 依赖模块

- \`moduleName\` - 模块用途说明
- \`anotherModule\` - 模块用途说明
```

## CHANGELOG.md 规范

### 基本结构

```markdown
# 更新日志

本文档记录 [模块名称] 模块的所有重要更新和变更。

## [未发布]

### 计划中
- 待添加的功能和改进

---

## [版本号] - YYYY-MM-DD

### 新增
- ✨ 新功能描述

### 改进
- 🔄 改进描述

### 修复
- 🐛 修复描述

---

## 更新说明

### 版本号规则
- **主版本号** (x.0.0): 不兼容的 API 变更
- **次版本号** (0.x.0): 新增功能，向后兼容
- **修订号** (0.0.x): 问题修复和小改进

### 更新类型标记
- ✨ **新增** - 新功能
- 🔄 **改进** - 功能改进或优化
- 🐛 **修复** - Bug 修复
- 📝 **文档** - 文档更新
- 🔧 **配置** - 配置变更
- ⚠️ **破坏性变更** - 不兼容的变更
- 🗑️ **废弃** - 即将移除的功能

---

*最后更新: YYYY-MM-DD*
```

### 详细规范

#### 版本记录格式

每个版本记录应包含：

1. **版本号和日期**: `## [x.y.z] - YYYY-MM-DD`
2. **更新分类**: 新增、改进、修复、文档等
3. **详细变更列表**: 使用 emoji + 描述格式
4. **API 变更说明**（如适用）

#### 版本号规则

遵循 [Semantic Versioning](https://semver.org/)：

| 版本变更 | 触发条件 |
|---------|---------|
| 主版本号 (x.0.0) | 不兼容的 API 变更 |
| 次版本号 (0.x.0) | 新增功能，向后兼容 |
| 修订号 (0.0.x) | Bug 修复和小改进 |

#### 更新类型 emoji

| Emoji | 类型 | 使用场景 |
|-------|------|---------|
| ✨ | 新增 | 新功能、新组件 |
| 🔄 | 改进 | 性能优化、重构 |
| 🐛 | 修复 | Bug 修复 |
| 📝 | 文档 | 文档更新 |
| 🔧 | 配置 | 配置文件变更 |
| ⚠️ | 破坏性变更 | 不兼容的 API 变更 |
| 🗑️ | 废弃 | 即将移除的功能 |
| 🚀 | 发布 | 正式发布 |
| 🔐 | 安全 | 安全相关修复 |

#### 编写原则

1. **按时间倒序排列** - 最新版本在最上方
2. **详细描述变更** - 说明变更的原因和影响
3. **标记破坏性变更** - 使用 ⚠️ 突出显示
4. **关联 Issue/PR** - 如适用，引用相关编号

## 检查清单

在提交模块文档前，请确认以下事项：

### README.md 检查

- [ ] 标题包含模块名和中文描述
- [ ] 创建/更新时间已填写
- [ ] 概述简洁明了
- [ ] 核心功能使用 emoji 格式
- [ ] 架构图展示了主要文件
- [ ] 配置字段有详细表格说明
- [ ] API 接口文档完整
- [ ] 提供了使用示例
- [ ] 注意事项列出了关键限制

### CHANGELOG.md 检查

- [ ] 版本号遵循语义化版本规范
- [ ] 每个版本有明确的日期
- [ ] 更新内容按类型分类
- [ ] 使用正确的 emoji 标记
- [ ] 破坏性变更已标注
- [ ] 最后更新时间已更新
